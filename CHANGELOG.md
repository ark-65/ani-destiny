# AniDestiny 更新日志

> 中文更新日志 | [English Changelog](./CHANGELOG_en.md)

## [Unreleased]

### ✨ 新增
- 播放进度滑块现在会向读屏软件播报当前定位时间与总时长，让视障用户在拖动进度时获得真实的时间反馈，而不是只有抽象百分比。

### 🐛 修复
- 为 HLS（m3u8）离线下载完成首阶段闭环：在解析并校验 master/media 清单后，`start()` 现在会下载分片、落地 `index.m3u8` 并将任务置为 `completed`；清单链路异常仍按 `invalidManifest` / `networkError` / `unknown` 失败，Live 清单仍为 `unsupported`，并补充分片下载与本地清单产出的回归覆盖。
- 补齐 HLS 离线闭环中的“重启可续传”切片：任务重试时会复用已落盘的分片文件，失败任务保留 `localPath` 以便恢复；补充回归测试验证仅下载缺失片段，并修复跨平台测试断言（以 `path.separator` 区分）。
- 完善闭环下一步：当离线下载完成并落盘时，下载卡片直接显示 `播放` 按钮，调用本地播放器入口打开本地文件；补充回归测试验证 completed 卡片点击播放按钮会触发离线播放动作。
- 增强 HLS 离线闭环完整性：在分片下载完成后新增落盘验证，确认 `index.m3u8` 依赖的每个分片文件存在且非空；若任一分片缺失或空文件则写 `invalidManifest` 失败，防止本地播放时出现“路径存在但内容损坏”的假成功。
- 规范播放器离线入口对 `file://` URL 的处理：在本地清单文件不存在时直接判定为“无可播放源”，避免误走加载链路，并补充 `test/player_page_test.dart` 覆盖本地文件存在/缺失两种离线播放前置校验。

## [1.0.6] - 2026-07-21

### ✨ 新增
- 新增“反馈问题”闭环：点击后会复制已脱敏的诊断摘要，并打开预填的 GitHub issue；如果用户没有登录 GitHub，也可以把同一份报告直接粘贴到其他反馈渠道。

### 🐛 修复
- 修复下载列表中“不支持离线保存”提示只有说明、没有恢复入口的问题：任务卡片现在会直接提供“查看下载线路”，并返回对应番剧与集数的源列表，保留原数据源上下文，让用户能立即切源后重试。
- 修复测试静态分析告警：将 `player_page_test.dart` 中 `SnackBar` 动作回调从可空调用改为直接调用，以消除 `flutter analyze` 的 `invalid_null_aware_operator` 失败。
- 修复下载动作失败场景的落点：不再把不支持下载类型误导到“Review in Downloads”，现在不支持离线保存的条目回到“查看下载线路”（携带该集 `focusEpisodeId` 与来源 `sourceId`），并直接跳回到对应集的源列表，形成更完整的恢复路径。
- 修复播放器无可播放源场景下的回归用例：同步 `player_page_test` 对应断言到当前 `sourceUnavailableSuggestion` 文案（回到该集源列表后切换源再重试），防止 CI 因文案漂移误报。
- 优化播放器无可播放源恢复链路：在错误面板加入“选择播放线路”入口，用户可直接跳回该集源列表确认可用线路，减少“看见建议不知下一步怎么做”的认知中断。
- 完善无可播放源恢复闭环：播放器返回详情页时携带 `focusEpisodeId`，详情页会高亮该集并保持“返回该集源列表后切源再重试”链路的可执行连续性，让用户落点更精准。
- 增强无可播放源恢复闭环回归覆盖：新增 `player_page_test` 测试，验证“选择播放线路”在无可播放源状态下会跳转到带 `sourceId` 和 `focusEpisodeId` 的源列表页面，避免跳转参数失配。
- 修复番剧详情在无可播放源时的恢复动作缺口：当 `No playable source found` 提示出现时，补充“选择播放线路”入口并在跳转中保留 `sourceId` 与 `focusEpisodeId`，让用户能一键回到该集源列表继续恢复。
- 继续减少源诊断噪音：`SourceFallbackServiceImpl` 不再在 fallback 诊断记录里保留工程化兜底 message，改为保持空 message 并保留可读原因，防止用户在诊断页只看到占位文本。
- 优化 `sourceUnavailableSuggestion` 文案为一条可执行单路径：回到该集源列表，切换到其他数据源后重试，减少用户在无可播放源场景里重复确认“下一步”的认知成本。
- 继续收敛缺源恢复引导文案：将 `sourceUnavailableSuggestion` 改为“先返回该集源列表再切换数据源、重试”的统一动作入口，减少无源场景下用户判断“下一步该做什么”的空窗期。
- 继续收敛反馈摘要中的 fallback 原因清洗：让反馈快照与运行/源诊断复用同一清洗路径，移除重复实现并补充回归，避免导出摘要里仍出现 `Source fallback used` 或 `Source attempt N` 技术噪音。
- 修复反馈快照源健康 `lastError` 字段仍会保留回退前缀（如 `Source fallback used`）的场景：统一从健康报错字段也走同一清洗路径，导出问题包里只保留可读原因。
- 进一步统一回退原因清洗路径：让运行诊断页的回退事件与消息正文都复用同一清洗规则，减少同一报文在设置页与诊断页之间出现的口径差异与噪音重现。
- 修复源诊断设置页与运行诊断清洗口径不一致：`Source Settings` 页面与其他诊断入口共用公共 fallback 原因清洗逻辑，避免同一类上游报文在不同入口重复出现 `Source fallback used / Fallback reason` 等工程前缀噪音。
- 修复 `diagnostic_sanitizer` 在遇到 `Exception:` / `AppException:` 这类异常前缀时仍将前缀保留给用户的问题：源诊断、运行诊断与复制反馈里统一按可读原因清洗后展示，并补充 `diagnostic_sanitizer_test.dart` / `anime_detail_page_test.dart` 回归。
- 修复番剧详情下载场景：当无可下载线路时，改用“无可下载源”提示文案，明确要求先回到当前集切源后重试，避免与无可播放源提示混淆。
- 修复无可下载源提示文案：当下载场景出现“未找到可下载线路”时，明确要求先返回该集再切换其他数据源后重试，减少用户在跳转与重试间迷失下一步。
- 优化缺源场景的共享提示文案：将 `sourceUnavailableSuggestion` 的“先回到该集顶部切源”式描述改为当前页面可直接执行的通用指引，让首页、搜索、番剧详情、时间表和播放器的无源提示更一致。
- 调整源回退服务输出：不再在 `message` 中返回 `Fallback reason:` 前缀，而是仅在有可读失败原因时追加原因文本，减少服务层文案里对用户的实现感噪音。
- 修复“上游源异常无法继续播放”提示：在文案中把“先切换到其他数据源再重试”与“先回到该集顶部切源后再重试”结合展示，减少用户在无源可播状态下的动作迷失。
- 修复下载回退提示的下一步：当番剧详情下载来自备用源时，新增“回到该集并切源重试”这一步提示，减少“目前只显示来源变化”导致的理解停顿。
- 优化播放失败文案，把“播放失败”场景的第一步明确为“先点重试恢复”，仅在未恢复时再建议切换线路，减少短时卡顿下的动作犹豫。
- 修复下载列表里“暂不支持离线保存”的任务说明口径不一致问题：未支持类型条目现在也会明确告知“先回到该集确认可直接下载源，再决定重试或移除”，与详情页下载入口保持同样的可执行下一步。
- 修复播放器与番剧详情“该下载仅支持留存查看”场景在 HLS / m3u8 文案上的回归：将测试断言同步到最新提示（先确认可下载源后再决定重试或移除），避免文案漂移再度拉低 CI 与产品一致性。
- 修复不支持离线保存下载的记录提示文案：明确下一步为“先回到该集确认可直接下载源，再决定重试或移除”，减少用户在遇到 HLS / m3u8 或 BT 记录时的误判。
- 修复源诊断、运行诊断与反馈摘要对 `Source fallback used · reason`（中点分隔）形态的兼容：当上游文案是 `Source fallback used · DNS timeout...` 时，不再保留前缀噪音，仍直接展示可读原因。
- 修复源诊断、运行诊断与反馈摘要里对 `Source fallback used =>` ASCII 箭头变体的兼容：当上游回退文案写成 `Source fallback used => reason` 或 `Source fallback used = reason` 时，不再把技术前缀暴露给用户，依然稳定展示可读原因。
- 修复 `Source fallback used / reason`（含全角变体 `｜`）这类分隔符形态下仍露出工程化前缀的问题：反馈摘要、运行诊断与源诊断都继续隐藏 `Source fallback used` 噪音并保留可读原因。
- 修复源诊断、运行诊断与反馈摘要在 `Source fallback used<reason>` 形态（角括号包裹原因）下仍露出实现前缀的问题：统一剥离 `<>` 包裹的后缀原因，让可读原因更完整。
- 修复 `test/feedback_package_collector_test.dart` 在 CI 的 `flutter analyze` 严格模式下因 `prefer_const_constructors` 3 条 info 被视为失败的问题：将测试中的 `SourceDiagnostic` 实例改为 `const` 构造，保持现网行为不变并恢复下载诊断快照回归链路。
- 修复下载清理文案里“待清理残留文件”表述不一致的问题：中文提示统一改为“标记为”表达，避免在手动清理路径里给用户制造术语歧义。
- 修复下载页残留清理回归断言与当前文案/交互脱节的问题：把单条手动清理残留场景的提示文案和“多条残留重检后仍有剩余条目”时顶部重检入口可见性的行为收束到现网实现，避免 CI 因旧期待文本误报。
- 修复下载列表批量清理在出现 `download_manual_cleanup_required` 失败时，未提供下一步复检入口的问题；现在会在该类失败存在时同步给出“重新检查”动作，单任务走任务级复检，多任务走批量复检，让用户在同一条反馈条里直接推进清理确认。
- 修复下载页单条任务移除失败时的下一步中断问题：当移除失败因 `download_manual_cleanup_required` 时，Snackbar 现在直接给出“重新检查”动作（可回到当前残留文件任务并触发重新检测），让用户无需猜测下一步，也不用先退出再手动寻找入口。
- 修复下载列表在仅剩一个“待手动清理残留文件”且存在多个可直接清理任务时，页面级“重新检查”入口会误触发跳转的问题；现在只保留任务卡片内的直接移除路径，避免额外跳级操作。
- 修复下载列表在恰好存在 1 个“待手动清理残留文件”与 1 个可直接清理任务时，页面级“重新检查”入口仍会消失的问题：保留该入口以便用户能在列表层面继续处理残留，减少定位到单张任务卡片才能重检的摩擦。
- 修复下载任务移除失败在报文仅携带纯字符串包装码（无 `AppException.code`）时仍看不到“重新检查”按钮的问题：只要报文里有 `download_manual_cleanup_required`，用户仍能在主路径上直接发起残留文件重检。
- 修复下载能力边界的可执行提示：不支持离线保存的下载记录现在会明确提示“先确认可直接下载源后再决定重试或移除”，减少用户在下载列表里误以为可直接下载的猜测。
- 修复下载动作失败提示在非 `AppException` 字符串报文中也能按 `download_*` 错误码解析本地化文案（如 `[download_busy]`、`download_network_error: ...`），减少网络/服务返回串直接透出“原始/工程化”提示的情况。
- 修复下载列表“批量清理已结束任务”在同时出现多类失败时只显示首条错误原因的问题：现在会汇总去重后的可读失败原因（例如“任务已不在列表”“待手动清理文件”）并与清理统计一起展示，让用户一次看到清理失败的真实差异。
- 修复下载列表批量清理失败场景文案收口：在 `clearEndedDownloadsPartialResult` 结果里保持“失败数量”与“可读失败原因”同时可见，避免仅凭数字让用户猜原因，并继续避免技术性异常文本（如 `StateError`）露出主流程。
- 修复下载动作失败后，仍会返回 `download_not_found` / `download_remove_not_allowed` / `download_manual_cleanup_required` 三类错误码的英文原始文案问题；现在改为本地化可读文案，避免用户在中文环境里看到技术措辞。
- 修复下载批量清理在 `removeEndedTask` 出现非 `AppException` 异常时仍只能看到“失败数量”；现在首个失败原因会按现有 `downloadActionErrorMessage` 路径兜底显示可读提示，并优先保持可执行的重试/下一步建议。
- 修复下载列表批量清理时，若遇到可解析动作异常，只显示“清理失败数量”不足以判断原因的问题；现在会在汇总文案下补一行可读失败原因（如“该下载任务不存在或已从列表移除”），让批量移除路径一次性讲清“发生了什么”。
- 修复下载动作因 `download_unsupported_type` 错误码时仍可能落到模糊文案的问题；现在会先展示“暂不支持该类型”并附带可重试建议，避免用户把“BT / 磁力/不支持类型”当作通用异常。
- 修复下载操作失败提示在 AppException 带有明确错误码（如网络/存储不可用）时，仍可能落到跨语境的“技术异常”口径的问题；现在会按错误码先落到本地化的可读原因，再接上可操作的重试提示，让下载失败提示先说清“为什么失败”，再告诉用户下一步。
- 修复下载动作在返回 `download_unexpected_error` 时，仍可能把底层实现感知带到用户界面；现在会改为“下载时发生意外异常 + 可重试建议”的本地化组合文案，避免下载失败反馈在主路径上突然泄露系统细节。
- 修复下载动作在 `download_busy` 错误码场景下仍可能落到非确定性原始文案；现在会使用本地化的“下载操作仍在进行中”提示，保持下载错误反馈在主路径上的稳定与可执行性。
- 修复下载页恢复到前台后同时还剩多条需要手动清理的残留文件、且仍有多条可直接清理任务时，SnackBar 现在也保留对应的“清理已结束任务”动作，让收尾流程更快闭环，不再要求用户先去列表里找入口。
- 修复下载页恢复到前台后仅剩“待清理残留文件”任务时，SnackBar 仍可能缺少可继续动作；现在会补上“重新检查”入口，直接触发当前残留任务重检，避免用户需要先回卡片才能继续收尾。
- 修复下载列表仅剩一条“待清理残留文件”任务且列表里无可直接清理任务时，下载页顶部不再无入口；现在会直接显示“重新检查”按钮，用户可在列表级发起重检，减少卡片内二次定位操作。
- 修复下载页在仍有“待清理残留文件”任务时，即使同时存在可直接清理的任务也会保留“重新检查”入口，让用户在列表层就能继续处理残留，不用先定位到特定任务卡片。
- 修复单任务“待手动清理残留文件”文案回归：当列表出现残留清理提示时，测试期望文案与当前页面文案一致，避免单条失败场景里仍出现旧版“return here”文案导致 CI 误报。
- 修复下载列表加载失败场景仍会透出原始英文文案的问题；现在即使发生异常也先按下载动作错误码映射成本地化提示，先说清“为什么失败”，再给用户稳定的重试反馈。
- 修复下载反馈文案在 `code` 缺失但 `AppException` 消息本身已携带 `[download_not_found]` 等已知错误码时，仍可能露出原始文本的问题；现在会优先提取并按已知码走本地化提示，保持错误信息仍对用户可读。
- 修复 `AppException` 消息中如果只写了 `[download_not_found]` 这类括号码（缺少 `AppException:` 前缀）时仍无法识别的问题，现可直接按错误码走本地化文案并隐藏实现细节，让同一条下载失败主路径在更多服务端报文变体下仍保持可读。
- 修复 `AppException` 变体如 `AppException [download_not_found] ...`（冒号缺失）仍被识别为未知文案的边界场景，现统一按错误码提取为本地化提示，避免下载失败反馈再次露出实现口吻。
- 修复下载任务卡片在未知异常兜底场景里仍展示英文“unexpected”类文本的问题；现在统一改为本地化失败文案（如“下载时发生意外异常”），让下载失败卡片在中文/英文/日文里都先读得懂，再决定重试。
- 修复源诊断与运行诊断页在 fallback 场景下还会出现“Source fallback used.”这类工程化底噪；现在有 fallback 原因时优先展示可读失败原因，不再在用户面前复读技术兜底文案。
