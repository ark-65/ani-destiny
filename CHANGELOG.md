## [Unreleased]

### 🐛 修复
- 修复下载页已经确认残留文件清理完成、提示里也说了“现在可以点上面的批量清理”，用户却还得自己再把视线拉回顶部找按钮的问题；现在这类 `重新检查` / 恢复提示会直接带上可点击的“从列表清理 N 个已结束任务”操作，让下载收尾能在结果出现的那一刻顺手完成。
- 修复下载页顶部已经同时出现批量“从列表清理 N 个已结束任务”入口、列表里却还剩最后 1 条 `待清理残留文件` 任务时，页面说明仍只催用户回头逐条 `重新检查` 的问题；现在这段说明会直接讲清楚“上面可以先清掉哪些已收尾任务”和“这条残留任务删完文件后该怎么收尾”，让最后一步不再像两条断开的路径。
- 修复下载页即使只剩 1 条可直接收尾的已结束任务时，顶部也继续摆着全局“从列表清理 1 个已结束任务”入口、让单条收尾看起来像批量动作的问题；现在 AniDestiny 只会在确实有多条可立即清理任务时才显示这个页面级入口，而单条任务会直接回到卡片内移除，让下载收尾更安静也更一致。
- 修复直链下载已经进入 `已停下` / `正在停下` 后，任务卡片仍继续挂着一条 `0%` 伪进度、像是还在等待继续下载的问题；现在 AniDestiny 会在这些停下态里收起旧进度，只保留真实的停下说明、重试入口和后续清理结果，让结束态先像结束态。
- 修复下载页已经能告诉用户哪些放弃下载仍留着残留文件、但复制出去的反馈摘要还没把这份残留文件的本地路径和下一步动作一起带上的问题；现在这类支持摘要会补上残留路径，并明确提示用户先删文件、再回到下载列表执行对应的 `重新检查` / 批量复查动作。
- 修复用户手动删掉最后 1 份下载残留文件后重新回到 AniDestiny、而页面上同时还有顶部批量“从列表清理 N 个已结束任务”入口时，恢复提示仍会用别扭的泛称来描述这条已经可收尾的任务问题；现在这类提示会直接说明你既可以点顶部批量清理，也可以立刻把这条任务从列表移除，让最后一步收尾更像一个明确动作。
- 修复用户手动删掉部分下载残留文件后，AniDestiny 虽然已经重新放出了顶部“从列表清理 N 个已结束任务”入口，却仍只催你继续处理最后 1 份残留、没顺手指出哪些任务现在已经可以直接收尾的问题；现在无论是返回应用后的恢复提示，还是顶部批量“重新检查残留文件”的反馈，都会同时说明“哪些任务可以立刻批量清理”以及“最后那份残留接下来该怎么处理”。
- 修复下载页只剩单条 `待清理残留文件` 任务时，用户点完这条任务的 `重新检查`、AniDestiny 也已经确认残留文件不在了，却还没顺手指出页面上已经重新出现的批量“从列表清理 N 个已结束任务”入口的问题；现在这类单条复查成功提示也会直接把用户带到顶部批量清理动作，让最后一步收尾不再绕回逐条判断。
- 修复用户只删掉了部分下载残留文件后，AniDestiny 虽然会告诉你“清掉了几份、还剩几份”，却还没把下一步动作顺手讲清楚的问题；现在无论是回到应用后的恢复提示，还是顶部批量“重新检查残留文件”的反馈，都会直接说明接下来该继续删哪几份、该点顶部批量复查还是对应任务的“重新检查”。
- 修复用户已经把多条下载残留文件手动删干净后，AniDestiny 虽然会确认“现在可以移除了”，却没顺手指出页面上已经重新出现的批量“从列表清理 N 个已结束任务”入口的问题；现在这类结果提示会直接把最后一步动作说出来，让多条残留文件的收尾不再靠猜。
- 修复用户点完“从列表清理已结束任务”后，若列表里还留着 `待清理残留文件` 任务，结果提示仍只会泛泛说“这些残留文件还在”、却不会像页面正文一样直接告诉用户下一步该点哪一个复查动作的问题；现在这条结果反馈会和页面说明保持同一套口径，单条残留会明确引导回该任务的“重新检查”，多条残留则直接指出顶部批量“重新检查 N 份残留文件”入口。
- 修复下载页已经提供顶部“重新检查 N 份残留文件”批量复查入口时，页面说明和单条任务提示却还继续把用户往逐条点“重新检查”上带的问题；现在多条残留文件任务会明确提示你可以直接用顶部批量复查，也可以逐条确认，让“手动删残留后怎么收尾”这条路径终于讲成同一个动作。
- 修复用户刚点下 `先停一下`、AniDestiny 其实还在取消请求并清理未完成文件时，下载卡片顶部却已经像彻底停稳一样直接写成“已停下”的问题；现在这段过渡会明确显示“正在停下”，等清理真正落定后，再回到可重试的已停下状态。
- 修复用户刚点下 `放弃这个下载`、AniDestiny 其实还在取消请求并清理残留文件时，下载卡片却已经像彻底收尾一样直接写成“已放弃、文件已清掉”的问题；现在这段过渡会明确显示“正在放弃”，等残留清理真的落定后，再给出最终的已放弃或待手动清理结果。
- 修复直链下载在用户刚点下 `先停一下` 或 `放弃这个下载` 后，底层取消与残留文件清理还没真正落定，任务却已经像完全收尾一样过早放出下一步动作的问题；现在 AniDestiny 会等这次下载尝试真正 settle 后，才结束这次停下/放弃操作，避免用户在旧文件还没收干净时就抢先重试或从列表移除。
- 修复下载页已经把残留文件任务明确标成 `待清理残留文件`，页面级批量清理说明却还继续把同一类任务叫成旧的“已取消”、让状态标签和页面引导又重新分叉的问题；现在这组说明会直接沿用 `待清理残留文件` 的口径，让用户在决定“先手动删残留，还是现在批量清理”时看到的是同一套真话。
- 修复下载列表里已经把可直接收尾的放弃任务明确写成 `已放弃`，复制出去的反馈摘要却还继续把同一类状态和原因写成通用 `已取消` 的问题；现在支持摘要里的下载状态计数和最新问题原因也会沿用同一套 `已放弃` 说法，让用户眼前看到的状态和带去反馈的信息保持同一层语义。
- 修复下载列表里已经明确用“放弃这个下载”和“这个下载已放弃”来描述结束态，但状态徽章仍只写成通用的“已取消”、让放弃态看起来像系统错误或工程状态的问题；现在这类任务会直接标成“已放弃”，让状态、说明和后续动作说的是同一件事。
- 修复用户已经 `放弃这个下载` 后，任务卡片仍继续挂着一条 `0%` 进度和进度文案、像是在等待继续下载的问题；现在 AniDestiny 会在放弃后的卡片里收起这层伪进度，只保留真实的清理结果、残留文件说明和收尾动作，让结束态先像结束态。
- 修复用户手动删掉一部分下载残留文件后重新回到 AniDestiny，下载页仍只会泛泛提示“有文件清掉了”、却不告诉用户到底清掉了几份、还剩几份没收尾的问题；现在恢复提示会直接带上数量，把“已经可以移除哪些任务”和“还有哪些仍需清理”在回到应用的第一时间说清楚。
- 修复下载列表里同时留下多条“待清理残留文件”任务时，用户仍得逐条点 `重新检查` 才知道哪些残留文件已经手动删掉的问题；现在页面顶部会提供一个带数量的批量复查入口，一次刷新所有残留文件状态，并直接告诉用户哪些任务已经可以从列表移除、哪些还需要继续清理。
- 修复用户在下载页点完“从列表清理已结束任务”后，如果列表里还留着那些必须先手动删残留文件的放弃下载任务，结果提示却只说“已清理几条”、没把“为什么还有任务留着”当场讲清楚的问题；现在 AniDestiny 会在批量清理结果里直接补一句说明：仍带着残留文件路径的任务会继续留在列表里，直到那份半截文件真的消失，让“这次清掉了什么、为什么还有东西没消失”在动作完成那一刻就说清楚。
- 修复下载页顶部“从列表清理已结束任务”在只会清掉其中一部分记录时仍写得像要一口气清完整个结束列表的问题；现在这个批量入口会直接带上本次实际可清理的数量，让用户点下前就知道这次会收掉几条任务，而那些仍需手动处理残留文件的记录会继续留在列表里。
- 修复下载页已经把“残留文件还得手动处理”的任务明确标成 `待清理残留文件`，复制出去的反馈摘要却还继续把同一条任务写成普通 `已取消` 的问题；现在支持摘要里的下载状态计数和最新问题也会沿用同一套 `待清理残留文件` 说法，让用户眼前看到的状态和带去反馈的信息保持一致。
- 修复用户已经按提示手动删掉放弃下载留下的残留文件、重新回到 AniDestiny 时，下载页虽然会悄悄恢复成可移除状态，却没明确告诉用户“现在可以收尾了”的问题；现在页面在检测到残留已清掉后会直接给出确认反馈，让这条手动清理链路真正落到一个明确结尾。
- 修复下载页里那些要求用户先手动删除残留文件的任务，在用户点下 `重新检查` 后仍只会静默刷新、既不明确说清“残留还在”还是“现在可以移除”的问题；现在 AniDestiny 会直接反馈这次检查结果，让手动清理的最后一步不再像按钮没反应。
- 修复那些已提示“请先手动删除残留文件”的下载任务，在用户确实删完文件后仍缺少一个顺手回到 AniDestiny 完成收尾的闭环问题；现在这类任务会提供 `重新检查` 入口，用户从系统文件管理器回来时页面也会自动重查残留文件状态，让 `待清理残留文件` 能及时恢复成可从列表移除或批量清理的真实状态。
- 修复用户已经按说明手动删掉那份放弃下载后的残留文件，下载列表却仍因为旧的 `本地路径` 记录把任务继续锁成 `待清理残留文件`、既不给 `从列表移除` 也不给批量清理入口的问题；现在 AniDestiny 会按残留文件是否真的还在来恢复可移除状态，让“先手动删残留，再把任务从列表收掉”真正闭环。
- 修复用户在直链下载仍处于进行中时刚点下 `放弃这个下载`，任务就会立刻显示成 `待清理残留文件`、像是已经需要手动收尾的问题；现在 AniDestiny 会先把这次放弃如实落成普通 `已取消`，只有自动清理残留文件确实失败后，才会再升级成需要手动处理的状态。
- 修复下载列表里那些“已放弃但仍留有残留文件”的任务，在最醒目的状态标签上仍只写 `已取消`、用户必须再读下面长说明才知道这条任务其实还没收尾的问题；现在这类任务会直接显示 `待清理残留文件`，让状态标签、手动清理说明和操作可见性先说同一种真话。
- 修复下载页已经明确说明“残留文件没清掉前，这条已取消任务要继续留在列表里”，单条任务卡片却还继续给出一个 `从列表移除` 入口的问题；现在这类任务会直接收起这个伪动作，只保留残留路径和手动清理说明，让页面规则和单条操作先说同一种真话。
- 修复下载列表里如果只剩“已取消但仍带着残留文件路径”的任务，顶部却还继续留下一个无法执行的“从列表清理已结束任务”按钮的问题；现在 AniDestiny 只会在确实还有可立即清理的已结束任务时才显示这个批量入口，避免把“先手动删残留文件”误做成像是按钮失灵。
- 修复下载列表里的 `已取消` 任务如果还保留着一条没清掉的残留文件路径，`从列表移除` 和 `从列表清理已结束任务` 仍可能把这条唯一线索一起删掉的问题；现在 AniDestiny 会先再次尝试清掉残留文件，清不掉就把这条任务继续留在列表里，并直接说明需要先手动处理那份残留文件。
- 修复用户已经 `放弃这个下载` 后，任务卡片却只剩一个静态 `已取消` 状态、没有明确说清未完成文件到底有没有清掉的问题；现在 AniDestiny 会直接说明放弃结果，若自动清理失败还会保留残留文件的本地路径，方便用户当场手动收尾。
- 修复下载卡片把 `先停一下` 和 `取消` 并排放在一起、却只解释前者会怎样、没明确说清后者会放弃未完成内容的问题；现在直链下载会把这两个动作一起讲清楚，并把 `取消` 明确标成放弃下载，让用户在停下、重试和彻底丢弃之间更容易当场做对选择。
- 修复下载页顶部 `从列表清理已结束任务` 已经能批量清理记录，却还没在页面级别明确说清“这里只清列表、不删已下载文件”的问题；现在只要列表里存在已完成任务，这个入口下方就会直接补一条说明，让用户在批量清理前先知道本地文件仍会留在设备上。
- 修复直链下载已经被用户 `先停一下` 或 `取消` 后，列表仍继续挂着旧进度、设备上也可能残留一份这次注定不会续传的半截文件的问题；现在 AniDestiny 会在这两种结束动作后立刻清空这次停下的伪进度，并尽快丢弃对应的临时文件，让“已停下 / 已取消”先像真实结束，而不是像还保留着一段可继续的离线现场。
- 修复下载页对 `移除` / `清理已结束任务` 的说法仍像会删掉本地文件、实际却只是清掉列表记录的问题；现在 AniDestiny 会明确说明这些动作只是从列表移除任务，而已下载文件会继续保留在设备上，让下载管理这条链路先把结果说清楚。
- 修复直链下载刚好完成时，陈旧的 `先停一下` 或 `取消` 动作仍可能把任务从 `已完成` 误写回 `已停下` / `已取消` 的问题；现在 AniDestiny 只允许下载任务沿着合理状态流转，避免竞态把已经完成的离线结果重新说成未完成，并补上服务级回归保护。
- 修复未完成的直链下载任务仍提前暴露 `本地路径`、看起来像已经留下可直接使用离线文件的问题；现在 AniDestiny 只会在真正下载完成后展示本地文件路径，让“已停下 / 失败 / 准备中”这些状态先表达当前事实，而不是暗示离线文件已经可用。
- 修复下载页把直链下载的“先停一下”仍表现得像真正可无缝继续的“暂停”问题；现在下载中任务会直接说明这次只能先停下、下次可能从头重试，已停下任务也会明确显示为“已停下”并提供 `重试`，让当前能力边界先说清楚，并补上对应组件回归保护。
- 修复直链下载任务已经停在旧进度后再次 `重试`，列表却还会先沿用上一次进度、像是在无缝续传的问题；现在每次重试都会先把任务重置回新的准备态，再从 `0%` 重新开始，并补上服务级回归保护，让“可能从头开始”不只是一句文案。
- 修复这条“已停下后重试会先清空旧进度”的回归保护仍绑死 Unix 路径分隔符、导致 Windows CI 误报失败的问题；现在断言改为真正跨平台，夜间分支能在本地与 Windows 验证同一条下载诚实性闭环。
- 修复复制出的反馈摘要还继续把这类直链任务写成 `Paused`、和下载页已经改好的“已停下 / 重试”说法分叉的问题；现在支持摘要里的下载状态计数也会沿用同一套 `已停下` 术语，让用户眼前看到的下载状态和带去反馈的信息保持一致。

## [1.0.4] - 2026-06-25

### 🐛 修复
- 修复设置页“运行诊断”、播放器内“播放诊断”和复制出的播放摘要在没有实际线路值时，仍继续展示 `Line: -` 这类空占位的问题；现在 AniDestiny 会直接收起这类无信息量的线路行，让用户先看到真实播放现场，再决定是否带走支持信息。
- 修复设置页“运行诊断”、播放器内“播放诊断”和复制出的播放摘要还会把 `未知 URL 类型`、空的 `请求头名称` 这类没有信息量的占位行继续展示出来的问题；现在 AniDestiny 只会保留真正有助于确认这次播放请求的细节，让支持信息先安静、再有用。
